# 角色定义
你是资深 strongSwan/IKEv2/IPsec 开发者。现在在仓库 pqgm-ipsec 中实现实验性协议 PQGM-IPSec。请严格遵守以下边界与约束（必须）：

【项目目标（不要偏离）】
- strongSwan 6.0+ 为基线；利用其已实现能力：IKE_INTERMEDIATE(RFC9242)、RFC9370(ADDKE框架)、IKE分片(RFC7383)、ML-KEM（标注“strongSwan6.0+已实现”，不要重复造轮子）。
- 我们需要实现/补齐的核心是：
  1) 国密能力：使用 GMSSL 3.1.3 作为底层密码库，以 strongSwan 插件形式集成（插件名 gmalg），至少提供 HMAC-SM3(PRF)、SM2 公钥加密/解密（供 SM2-KEM 封装/解封装使用）、必要的 SM3/SM4 支持（按需要最小集）。
  2) 国密SM2双证书的分发，当协商使用SM2-KEM时，在中间交换开始阶段进行（SM2-KEM前）进行双证书的分发。签名证书用来做提前认证的DOS门控（不用真验证，模拟验证通过/不通过就行）；签名证书用来分发对端加密公钥
  3) 私有 KE method：实现 sm2-kem（DH Group Transform Type=4，私有 group=65001），作为 RFC9370 的 ADDKE1，完成 1 RTT 双向封装：
     - Initiator 生成 r_i，Responder 生成 r_r，r_len=32B（因为 PRF=HMAC-SM3 输出 32B）。
     - SK_sm2 = r_i || r_r（固定顺序，先 i 后 r），不做 KDF。
  4) swanctl.conf 配置扩展：新增字段指定签名证书与加密证书路径（例如 local.sign_cert / local.enc_cert），SM2-KEM 必须使用 EncCert 公钥。
  5) 认证：IKE_AUTH 强制 ML-DSA-65（FIPS 204）。如果 strongSwan 主线缺失 ML-DSA，允许以插件/补丁方式接入（优先插件化）。

【固定握手时序（必须严格一致）】
- 3 个 IKE_INTERMEDIATE，ADDKE 顺序：ADDKE1=SM2-KEM，ADDKE2=ML-KEM-768，证书在 ML-KEM 前发送：
  1) IKE_SA_INIT: SA(主KE=x25519, ADDKE1=sm2-kem[65001], ADDKE2=ml-kem-768, PRF=HMAC-SM3), KEi,Ni + N(INTERMEDIATE_SUPPORTED)
  2) IKE_INTERMEDIATE #1: 仅传双证书（CERT SignCert + CERT EncCert），本轮不携带 KE，不触发 RFC9370 密钥更新
  3) IKE_INTERMEDIATE #2: ADDKE1=SM2-KEM，双方交换 ct_i/ct_r，得到 SK(1)=r_i||r_r，更新 SKEYSEED(1), SK_*(1)
  4) IKE_INTERMEDIATE #3: ADDKE2=ML-KEM-768，得到 SK(2)=mlkem_ss，更新 SKEYSEED(2), SK_*(2)
  5) IKE_AUTH: 强制 ML-DSA-65 认证，并启用 IntAuth 绑定所有 intermediate（篡改必须导致失败）

【证书验证策略（实验简化，但要可追溯）】
- 证书链验证允许 bypass/accept-all（为了先跑通），但必须：
  - 证书解析/分类（SignCert vs EncCert）必须成功，否则失败
  - 日志明确记录“证书验证 bypass 已启用”
  - 依赖 IntAuth + 负例用例证明篡改会失败
